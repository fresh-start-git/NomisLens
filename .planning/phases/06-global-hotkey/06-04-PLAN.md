  <files></files>

  </read_first>
  <what-built>
    - HotkeyManager.start() returns False when RegisterHotKey fails with ERROR_HOTKEY_ALREADY_REGISTERED (1409)
    - Graceful stdout message points user to config.json
    - App continues running without the hotkey
    - BubbleWindow.destroy() calls _hotkey_manager.stop() which posts WM_QUIT; worker thread's finally block calls UnregisterHotKey
  </what-built>
  <how-to-verify>
    Part A (double-register graceful failure):
    1. Launch one instance: `python main.py`. Confirm [hotkey] registered line appears.
    2. In a SECOND terminal, launch another instance: `python main.py`.
    3. The second instance should print (approximately):
       `[hotkey] registration failed: another app is already using this combination (GetLastError=1409). The bubble will still run, but will not respond to the hotkey. Edit config.json to change the hotkey and relaunch.`
    4. AND also `[hotkey] continuing without hotkey support`
    5. The second instance's BUBBLE should still appear (graceful continuation).
    6. The hotkey, when pressed, should toggle only the FIRST instance's bubble (not the second).
    7. Close both instances.

    Part B (clean unregister on exit):
    1. Launch one instance.
    2. Press the hotkey — confirm it works.
    3. Close the instance cleanly (Alt+F4 or normal close path).
    4. Immediately relaunch: `python main.py`.
    5. The new instance's [hotkey] registered line should show SUCCESS (not 1409).  This proves the previous instance's destroy() called UnregisterHotKey before the process exited.
    6. If you see 1409 on relaunch, the previous instance LEAKED its registration — that's a Pitfall 2 regression.

    Part C (--no-hotkey escape hatch):
    1. Close any running instances.
    2. Launch: `python main.py --no-hotkey`.
    3. Expected stdout: `[hotkey] disabled by --no-hotkey flag`.
    4. Bubble should appear normally.
    5. Press the configured hotkey — bubble should NOT toggle (no handler).
    6. Close instance.
  </how-to-verify>
